## 一、使用建议

### 1.1 这份题库怎么用？

- 第一轮：先把两个项目各自讲顺，确保 3 分钟项目介绍能流畅输出
- 第二轮：按专题刷基础题，优先 JVM、并发、MySQL、Redis、网络
- 第三轮：按“项目题 + 基础追问”串讲，训练从项目过渡到八股
- 第四轮：补算法高频题，至少保证能稳定写出常见中等题

### 1.2 优先级建议

- `P0`：项目介绍、MySQL、Redis、JVM、并发、HTTP/TCP
- `P1`：Spring、Dubbo/RPC、Netty、Linux 排障、设计题
- `P2`：消息队列、分布式事务、Kubernetes、可观测性

## 二、项目题总览

### 2.1 项目 A：Lumina-RPC

建议主讲方向：

- 这是一个自研 RPC 框架项目，重点体现协议设计、网络通信、服务治理和扩展机制
- 核心亮点包括自定义协议、Netty 通信、动态代理、服务注册发现、负载均衡、容错、SPI 扩展
- 如果面试官偏底层，这个项目是你的主力项目

重点题目：

1. 请介绍一下 Lumina-RPC 的整体架构
2. 你的 RPC 协议为什么这样设计？消息头字段分别有什么作用？
3. Netty 为什么适合做 RPC 框架？线程模型是什么？
4. 粘包拆包是怎么处理的？
5. 动态代理是怎么让远程调用看起来像本地调用的？
6. 为什么选 ByteBuddy？和 JDK 动态代理、CGLIB 比较呢？
7. 服务注册发现流程是怎样的？
8. 负载均衡策略有哪些？一致性哈希怎么实现？
9. Failover、Failfast、Failsafe、Forking 分别适合什么场景？
10. 熔断和限流分别解决什么问题？怎么配合？
11. SPI 是怎么设计的？为什么需要 SPI？
12. 超时控制在哪几层做了？各自作用是什么？
13. 如何实现优雅停机？
14. 如何排查 RPC 调用超时？
15. 如果让你继续做这个 RPC 框架，你会优先加什么能力？

进阶追问：

- 请求响应怎么做关联？
- Channel 池怎么保证线程安全？
- 跨线程如何传递 TraceId？
- 如果要支持跨语言调用，协议要怎么改？
- 如果让你支持百万 QPS，你会改哪些点？

### 2.2 项目 B：AI 零代码应用生成平台

建议主讲方向：

- 这是一个 AI 驱动的代码生成平台，重点体现微服务拆分、SSE 流式输出、AI Tool Calling、部署链路
- 核心亮点包括 Dubbo 微服务、LangChain4j Agent、多轮对话生成、预览与部署、截图与源码下载
- 如果面试官偏业务落地和工程化，这个项目是你的主力项目

重点题目：

1. 请简单介绍一下这个项目是做什么的
2. 为什么选择微服务架构？为什么拆成 user/app/ai/screenshot？
3. 项目的核心业务流程是什么？
4. AI 生成链路是怎么跑通的？
5. 什么是 Tool Calling？项目里怎么防止工具无限调用？
6. 智能路由是怎么选代码生成类型的？
7. 为什么前端用 SSE？和 WebSocket 有什么区别？
8. 流式输出时怎么保存完整 AI 响应？
9. Redis 在项目里有哪些用途？
10. 限流怎么实现？为什么选 Redisson？
11. Session 在微服务场景里怎么共享？
12. Dubbo 在项目里怎么用？和 Feign 有什么区别？
13. Triple 协议是什么？
14. Vue 项目生成后如何预览、构建、部署？
15. 如何保证 AI 生成代码的安全性和质量？

进阶追问：

- 如果 AI 响应很慢怎么优化？
- 如果用户量暴涨 10 倍，瓶颈会在哪？
- 如果生成的是恶意代码怎么办？
- 多轮对话上下文怎么控制长度？

### 2.3 项目组合回答模板

当面试官问“你做过最有挑战的项目是什么”时，可以这样组织：

1. 先讲 AI 平台，突出业务复杂度、完整链路、微服务、AI 编排
2. 再补充 RPC 项目，突出底层原理、框架设计、网络通信
3. 最后总结：
   - 一个偏业务落地
   - 一个偏基础设施
   - 共同体现你的工程能力和底层理解

### 20.1 怎么回答项目介绍，才不会像背八股？

回答思路：

- 先说项目解决什么问题，不要一上来背技术栈
- 再说核心链路，让面试官快速知道系统是怎么跑起来的
- 最后说你做了什么、难点是什么

示例回答：

“我有两个比较能代表我的项目。一个是 AI 零代码应用生成平台，用户输入自然语言需求，系统会通过 AI 生成前端应用代码，支持流式预览、二次编辑、部署和下载源码。这个项目更偏业务落地和工程化。另一个是自研 RPC 框架，重点在协议设计、Netty 通信、服务发现、负载均衡和容错，这个项目更偏底层原理。前者体现我做复杂业务系统的能力，后者体现我对底层框架和网络通信的理解。”

发散点：

- 如果面试官打断你，优先展开“核心链路”和“你负责的部分”
- 如果面试官偏基础，立刻切到 RPC 项目
- 如果面试官偏业务和落地，立刻切到 AI 平台

### 20.2 你做过最难的项目是什么？

回答思路：

- 别说“都挺难的”
- 选一个最能体现复杂度的项目
- 难点最好体现“技术 + 取舍 + 故障处理”

示例回答：

“对我来说更有挑战的是 AI 零代码平台。它的难点不只是调通大模型，而是把生成、预览、修改、部署这一整条链路串起来。比如生成阶段要处理 SSE 流式输出和上下文记忆，预览阶段要把生成结果落盘并映射成静态资源，部署阶段又要考虑构建、目录隔离和截图回填。这里面很多问题都不是单点技术问题，而是多个模块之间的衔接问题。我在这个项目里最大的收获是开始从‘写一个接口’转向‘把一个完整流程稳定跑起来’。”

发散点：

- 下一句就可以接“其中一个具体难点是……”
- 最好自己准备一个“最难点”的展开版本，控制在 2 分钟内

### 20.3 为什么拆微服务，而不是单体？

回答思路：

- 不要空谈“大厂都这么做”
- 结合你项目的业务边界、资源特性、依赖差异来讲
- 一定要承认微服务的成本

示例回答：

“这个项目拆微服务，不是为了追潮流，而是因为几个模块的边界和运行特征差别比较大。比如用户服务更偏常规 CRUD，AI 服务偏外部模型调用和对话编排，截图服务依赖浏览器运行环境，构建服务甚至直接用了 Node。把它们拆开后，部署依赖、资源需求和故障隔离都更清晰。缺点也很明显，就是链路更长、排查更复杂、配置管理成本更高。所以如果是一个非常早期、需求还不稳定的项目，我其实会先偏向单体，等边界稳定再拆。”

发散点：

- 可以顺带说“这个仓库实际上就保留了单体到微服务的演进痕迹”
- 面试官很喜欢听“不是所有场景都适合微服务”

### 20.4 为什么选 Dubbo，不选 Feign？

回答思路：

- 不要答成“Dubbo 性能更高”就结束
- 要说协议、治理能力、适用场景
- 最后补一句“如果是纯 HTTP 生态也可以选 Feign”

示例回答：

“我在这个项目里选 Dubbo，主要是因为服务间调用是内部调用，关注点更偏性能和治理。Dubbo 在服务注册发现、负载均衡、超时重试这些能力上比较成熟，而且 Triple 协议也比较现代。Feign 更适合 HTTP 风格明显、和网关生态结合更紧的场景。如果系统内部服务比较少、接口风格偏 REST，我觉得 Feign 也完全合理。这个项目选 Dubbo，本质上是因为它更贴合内部服务通信的场景，而不是说 Feign 不好。”

发散点：

- 面试官可能追问 Triple、序列化、重试幂等等
- 可以顺着切到 RPC 项目，体现你不是只会用框架

### 20.5 为什么前端用 SSE，而不是 WebSocket？

回答思路：

- 先说业务模式，再说协议特征
- 说明 SSE 是“够用且更简单”

示例回答：

“在这个项目里，前端和后端的交互模式本质上是‘前端发起一次生成请求，后端持续把生成结果往前推’，这是典型的单向流式输出场景。所以 SSE 很适合，浏览器原生支持也比较好，实现成本比 WebSocket 低。WebSocket 更适合强双工场景，比如多人协同编辑、在线聊天、实时游戏这种。这里如果为了流式输出上 WebSocket，其实有点过度设计。”

发散点：

- 可以补充“断线重连和结束标记是怎么做的”
- 也可以提“如果未来支持多人协同编辑，我会考虑 WebSocket”

### 20.6 Redis 在项目里怎么用？别只说缓存

回答思路：

- 按场景分，不按功能名分
- 至少说缓存、Session、限流、对话记忆

示例回答：

“这个项目里 Redis 不是单纯做热点缓存。第一，它承接了 Session 共享，解决微服务下登录态一致的问题。第二，它做了一些缓存场景，比如精选应用列表，减轻数据库压力。第三，它参与限流，避免 AI 对话接口被短时间打爆。第四，在 AI 这条链路里，它还承担了对话记忆存储的角色。也就是说 Redis 在这里既是性能组件，也是状态协调组件。”

发散点：

- 继续追问就切到缓存一致性、缓存三大问题、分布式锁

### 20.7 缓存和数据库一致性怎么回答更像工程实践？

回答思路：

- 别直接背“先写库再删缓存”
- 先承认“强一致很难，工程上追求最终一致”

示例回答：

“大多数业务场景下，我会优先追求最终一致而不是绝对强一致。比较常见的做法是先更新数据库，再删除缓存，因为删缓存的幂等性更好。如果并发很高、又很怕旧值回灌，我会考虑延时双删或者通过 MQ 做异步失效。再往上走，如果是特别敏感的数据，就应该减少缓存依赖，甚至改成读主库。我的理解是，一致性方案不是背模板，而是看数据的重要程度、脏读容忍度和流量规模。”

发散点：

- 面试官如果继续问，你可以举“精选应用列表”和“用户信息”区别处理

### 20.8 如何回答 MySQL 索引设计，不显得只会背定义？

回答思路：

- 从查询场景出发
- 说出“为什么建这个索引”，而不是只说“建联合索引”

示例回答：

“我设计索引时不会先想技术名词，而是先看系统里最常见的查询路径。比如如果后台列表页经常按 `userId + createTime` 查应用，那我会优先考虑围绕这个路径建联合索引，因为它既能过滤用户，又能支持排序。如果查询只取少数字段，还可以进一步考虑覆盖索引，减少回表。反过来说，如果某个字段区分度很低，比如性别、状态位，单独建索引就没太大意义。索引的核心不是多建，而是尽量贴着高频查询路径建。”

发散点：

- 如果面试官追问索引失效，就补 `%like`、函数、隐式类型转换、范围查询后断裂

### 20.9 怎么回答慢 SQL 排查？

回答思路：

- 讲排查顺序，而不是只说 `explain`
- 要体现“先定位，再优化”

示例回答：

“我一般会先确认是不是数据库层的问题，而不是直接怀疑 SQL 本身。排查顺序通常是：先看接口 RT 和数据库监控，确认是不是 SQL 慢；再抓慢日志，看具体是哪条语句；然后用 `EXPLAIN` 分析执行计划，重点看有没有走索引、有没有回表、扫描行数是不是太大；如果 SQL 本身没问题，再看是不是锁等待、连接池打满或者磁盘 IO 抖动。因为慢 SQL 不一定是 SQL 写得差，也可能是并发、锁冲突或者资源不足导致的。”

发散点：

- 面试官很喜欢你说“不是一上来改 SQL”

### 20.10 synchronized、volatile、CAS 怎么串起来回答？

回答思路：

- 别分三道题背
- 用“可见性、原子性、有序性”串起来

示例回答：

“我理解这几个关键字最好的方式，是从并发的三个核心问题看。`volatile` 主要解决可见性和一定程度的有序性问题，但不保证复合操作的原子性。`synchronized` 是更重的同步手段，它既能保证互斥，也能保证可见性。CAS 更像一种无锁乐观更新方式，适合冲突没那么高的场景，但要注意 ABA 问题。所以这几个不是谁替代谁，而是在不同冲突程度和性能要求下的不同选择。”

发散点：

- 继续追问就说 AQS、ReentrantLock、原子类

### 20.11 线程池参数怎么回答更稳？

回答思路：

- 不要直接背 7 个参数名
- 先讲为什么需要线程池，再讲关键参数怎么权衡

示例回答：

“线程池的核心不是记住参数，而是控制系统并发度，避免线程无限创建把机器拖死。我最关注的其实是 4 个点：核心线程数、最大线程数、阻塞队列和拒绝策略。CPU 密集型任务一般线程数接近 CPU 核数，IO 密集型可以适当放大。队列不能无脑设太大，不然问题会从‘线程太多’变成‘请求堆积太久’。拒绝策略也不是随便选，我更倾向于根据业务决定是快速失败、降级，还是把任务回退给调用方执行。”

发散点：

- 可以顺着讲 AI 调用、截图、构建这些是否适合线程池隔离

### 20.12 JVM 怎么回答，才不会像背内存图？

回答思路：

- 把 JVM 和排障联系起来
- 面试官更想听你会不会用，不只是记得住

示例回答：

“如果只是背 JVM 内存结构，面试价值其实不高。我更习惯把 JVM 放到实际问题里理解。比如接口 RT 飙升，我会想是不是 GC 频繁；系统 OOM，我会判断是堆内存不够、对象泄漏，还是线程太多把栈打爆；服务启动异常，我会想到类加载和依赖冲突。也就是说 JVM 对我来说不只是知识点，而是理解 Java 服务运行时行为的一套视角。”

发散点：

- 后续可以具体展开内存区、GC、类加载

### 20.13 Full GC 频繁怎么排查？

回答思路：

- 别一上来调参数
- 先确认“是正常业务量带来的，还是泄漏/配置问题”

示例回答：

“如果 Full GC 频繁，我会先看是短时间流量激增导致对象分配过快，还是内存真的回不下来。排查上一般先看 GC 日志和监控，确认 Full GC 频率、停顿时间和堆使用趋势；如果 Full GC 后老年代还是很高，我会怀疑内存泄漏，再配合堆 dump 分析大对象和引用链；如果回收后能降下来，但很快又涨满，可能是对象创建速率过高或者堆配置太小。最后才是做参数调优，因为参数优化解决不了代码泄漏。”

发散点：

- 补一句“先定位，再调优”，面试官印象会很好

### 20.14 为什么 Redis 单线程还这么快？

回答思路：

- 不要只说“纯内存”
- 说完整：内存、IO 多路复用、数据结构、避免线程切换

示例回答：

“Redis 快不是因为‘单线程’，而是因为它用单线程避免了锁竞争和线程切换，同时又利用内存操作和高效数据结构把单次命令做得很轻，再加上 IO 多路复用把网络层效率提上来。也就是说，单线程只是结果，不是根本原因。根本原因是它的工作负载适合这种模型：命令短、小而快、状态集中在内存里。”

发散点：

- 如果继续追问，可以说 Redis 6 开始网络读写也做了多线程优化

### 20.15 怎么回答 Session、JWT、Token 的区别？

回答思路：

- 先说本质，再说适用场景
- 别把 Token 和 JWT 混为一谈

示例回答：

“Session 本质上是服务端保存状态，客户端只带一个会话标识，所以更容易集中控制和失效；JWT 是一种自包含的令牌格式，用户信息和签名都放在令牌里，更适合分布式和无状态场景，但撤销会更麻烦；Token 是更宽泛的概念，JWT 只是 Token 的一种。这个项目里用 Session 主要是因为后台管理和用户系统更偏传统 Web 场景，而且已经配了 Redis 做共享，成本比较低。”

发散点：

- 可以继续说 CSRF、登出失效、权限粒度控制

### 20.16 HTTPS 为什么安全？

回答思路：

- 不要只答“因为加密”
- 至少说机密性、完整性、身份认证

示例回答：

“HTTPS 的安全性主要体现在三件事上。第一是机密性，传输内容经过对称加密，别人抓包也看不懂。第二是完整性，消息传输过程中如果被篡改，校验会失败。第三是身份认证，证书体系让客户端能确认自己连的是目标服务，而不是中间人。所以 HTTPS 不只是‘加了一层 SSL’，而是同时解决了窃听、篡改和冒充三个问题。”

发散点：

- 如果被追问握手过程，讲非对称交换密钥 + 对称加密传输即可

### 20.17 如何回答“AI 生成代码不稳定怎么办”？

回答思路：

- 这是很好的开放题，要体现工程思维
- 不要只说“换模型”

示例回答：

“AI 生成不稳定，本质上不是单个 Prompt 的问题，而是整个生成链路缺少约束和反馈。我会从几个层面做：第一，输入侧收敛需求，比如结构化提示和模板化约束；第二，生成侧控制输出格式，尽量做结构化，而不是纯自由文本；第三，增加后处理，比如代码解析、静态检查、构建验证；第四，对失败场景做重试、降级和兜底。我的理解是，AI 在工程里不能当‘神谕’，而要当一个可控但不完全可信的组件来接入。”

发散点：

- 可以补充 guardrail、tool 限次、构建校验、回滚策略

### 20.18 如果流量突然涨 10 倍，你先看哪里？

回答思路：

- 不要大而全
- 先抓主链路瓶颈：入口、缓存、数据库、外部依赖、异步能力

示例回答：

“我会先看主链路中最贵的环节，而不是平均发力。对这个 AI 平台来说，首先看 AI API 调用是不是扛得住，因为那通常是最慢也最贵的依赖；其次看 SSE 长连接和应用服务线程资源；再看 Redis 和 MySQL 有没有因为热点请求或列表查询被打穿；最后看构建和截图这类重任务是不是需要异步削峰。我一般会优先通过限流、缓存、任务隔离和异步化先把系统稳住，再考虑横向扩容。”

发散点：

- 这题核心是“先保命，再优化”

### 20.19 如何回答“如果让你重做，你会改什么”？

回答思路：

- 不要说“没什么想改的”
- 也不要把现有方案全盘否定

示例回答：

“如果让我重做，我不会推翻现有架构，但会优先补几个方向。第一是可观测性，我会更早把日志、指标和链路追踪统一起来，因为微服务和 AI 链路都很依赖排障能力。第二是异步化，把构建、截图、部分后处理做成任务化，降低主链路阻塞。第三是质量保障，比如生成结果的自动校验、静态检查和回归测试。现有方案已经能跑通闭环，但如果往更稳定的生产系统走，这三块我觉得是最值得补的。”

发散点：

- 这题特别适合体现你的成熟度和反思能力

### 20.20 如何回答“你和别人相比的优势是什么”？

回答思路：

- 不要说“学习能力强、沟通能力好”这种空话
- 用项目结构来支撑你的优势

示例回答：

“我觉得我的优势是，既做过偏业务落地的系统，也做过偏底层原理的项目。像 AI 零代码平台这种项目，更考验我把复杂流程串起来、把系统真正跑通；而 RPC 框架这种项目，又训练了我对协议、网络、并发和框架设计的理解。所以我不是只会调接口，也不是只会背原理，我更希望自己能把原理和工程实践连起来。”

发散点：

- 最后可以落到“比较适合 Java 后端实习，需要既能写业务也愿意钻底层的岗位”

### 20.21 基础题别背定义，要怎么答？

通用模板：

1. 先说它解决什么问题
2. 再说核心机制
3. 再说适用场景
4. 最后说边界或代价

示例：

- 不要只说“Redis 快，因为内存”
- 要说“Redis 快，是因为它的命令模型、内存访问、IO 多路复用和线程模型共同决定的”

- 不要只说“索引能加快查询”
- 要说“索引本质上是用空间换时间，但写入、维护和回表也都有成本，所以要贴着查询路径建”

- 不要只说“线程池提高性能”
- 要说“线程池本质上是在控制资源边界，如果参数配错，它也可能把系统拖垮”

### 20.22 面试里的发散性思维，具体体现在哪？

你回答时，尽量主动补下面这几类信息：

- 设计原因：
  为什么这样做，而不是别的方案？

- 适用边界：
  这个方案在哪些场景好用，在哪些场景不适合？

- 代价与风险：
  它解决了什么问题，又引入了什么成本？

- 演进方向：
  当前这样做够用，如果规模扩大，你会怎么演进？

- 真实排障：
  如果真的出问题，你会先查什么，而不是先改代码

一句话总结：

“实习面试里，真正拉开差距的不是你会不会背知识点，而是你能不能把知识点讲成一个真实工程判断。”
