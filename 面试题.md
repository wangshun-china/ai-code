### 1.1 请简单介绍一下这个项目是做什么的？

**标准回答**：

这是一个 **AI 驱动的零代码应用生成平台**，核心功能是让用户通过自然语言描述需求，AI 自动生成完整的前端应用代码。

**项目背景**：
- 解决前端开发门槛高的问题，非技术人员也能快速生成网页
- 降低原型验证周期，从数天缩短到数分钟
- 适用于个人项目、落地页、后台管理系统等场景

**核心功能**：
1. **智能代码生成**：输入"帮我做一个登录页面"，AI 自动生成 HTML/Vue 代码
2. **多轮对话修改**：可以继续说"把按钮改成红色"，AI 会基于上下文修改
3. **实时预览**：代码生成后立即预览效果
4. **一键部署**：自动构建、部署、生成访问链接
5. **源码下载**：支持下载完整项目代码

**技术栈概览**：
- 后端：Java 21 + Spring Boot 3.x + Dubbo + Nacos + MyBatis-Flex + Redis
- 前端：Vue 3 + TypeScript + Vite + Ant Design Vue
- AI：LangChain4j + 通义千问/DeepSeek
- 部署：Docker + Nginx

### 1.3 为什么选择微服务架构？有什么优缺点？

**问题分析**：架构选型是考察重点，要能客观分析利弊，展现架构思维。

**标准回答**：

**选择微服务的原因**：

1. **业务边界清晰**：用户管理、应用管理、截图功能各自独立，职责分明
2. **技术栈灵活**：截图服务用 Java + Selenium，构建服务用 Node.js
3. **独立扩展**：AI 代码生成是 CPU 密集型，截图是 IO 密集型，可按需扩容
4. **故障隔离**：截图服务崩溃不影响核心的代码生成功能

**微服务的优点**：
| 优点 | 说明 |
|------|------|
| 独立部署 | 单个服务更新不需要重启整个系统 |
| 技术异构 | 不同服务可以用不同语言实现 |
| 弹性扩展 | 可以只对高负载服务扩容 |
| 故障隔离 | 单服务故障不影响其他服务 |

**微服务的缺点**：
| 缺点 | 说明 | 本项目应对 |
|------|------|------------|
| 运维复杂 | 需要管理多个服务 | 使用 Docker Compose 统一编排 |
| 分布式事务 | 跨服务数据一致性 | 目前业务简单，无强一致性需求 |
| 服务间调用 | 网络延迟和失败处理 | Dubbo 内置重试和容错机制 |
| 链路追踪 | 问题定位困难 | 计划引入 SkyWalking |

**对于实习项目的建议**：
如果团队规模小于 5 人，初期可以用单体架构，随着业务增长再拆分。本项目支持两种启动模式，体现了架构演进的思想。

### 1.4 项目中有哪些核心业务流程？

**问题分析**：理解业务流程是开发的基础，要能清晰描述核心场景。

**标准回答**：

**核心流程一：用户注册登录**

```
用户输入账号密码 → 参数校验 → 密码加密(MD5+盐) → 存入数据库 → Session 记录登录态
```

相关代码 `UserServiceImpl.java`：
```java
public long userRegister(String userAccount, String userPassword, String checkPassword) {
    // 1. 校验参数
    if (StrUtil.hasBlank(userAccount, userPassword, checkPassword)) {
        throw new BusinessException(ErrorCode.PARAMS_ERROR, "参数为空");
    }
    if (userAccount.length() < 4) {
        throw new BusinessException(ErrorCode.PARAMS_ERROR, "账号长度过短");
    }
    if (userPassword.length() < 8) {
        throw new BusinessException(ErrorCode.PARAMS_ERROR, "密码长度过短");
    }
    // 2. 查询用户是否已存在
    QueryWrapper queryWrapper = new QueryWrapper();
    queryWrapper.eq("userAccount", userAccount);
    long count = this.mapper.selectCountByQuery(queryWrapper);
    if (count > 0) {
        throw new BusinessException(ErrorCode.PARAMS_ERROR, "账号重复");
    }
    // 3. 加密密码
    String encryptPassword = getEncryptPassword(userPassword);
    // 4. 插入数据库
    User user = new User();
    user.setUserAccount(userAccount);
    user.setUserPassword(encryptPassword);
    this.save(user);
    return user.getId();
}
```

### 1.5 【追问】这个项目的技术选型有什么考量？

**问题分析**：技术选型能力体现架构思维，要能说明选择的原因和权衡。

**标准回答**：

**技术选型对比分析**：

| 技术点 | 选择 | 备选方案 | 选择理由 |
|--------|------|----------|----------|
| 后端框架 | Spring Boot 3.x | Spring Boot 2.x / Quarkus | 最新稳定版，支持 Java 21 虚拟线程，性能更好 |
| RPC 框架 | Dubbo 3.x | Feign / gRPC | 高性能、完善服务治理、Triple 协议支持流式 |
| ORM 框架 | MyBatis-Flex | MyBatis-Plus / JPA | 更轻量、原生支持多表关联、SQL 构建更灵活 |
| AI 框架 | LangChain4j | Spring AI / 原生 HTTP | Java 生态最成熟、社区活跃、文档完善 |
| 前端框架 | Vue 3 | React / Angular | 学习曲线平缓、Composition API 更灵活 |
| 缓存 | Redis + Caffeine | 仅 Redis | 多级缓存减少网络开销，本地缓存提升读性能 |

**关键决策点**：

**1. 为什么选择 Dubbo 而不是 Feign？**

```
Feign 优势：
- Spring Cloud 原生支持，开箱即用
- 声明式调用，代码简洁

Dubbo 优势：
- 性能更高：长连接 + 二进制协议
- 服务治理完善：内置限流、熔断、降级
- Triple 协议：基于 HTTP/2，支持流式传输

本项目选择 Dubbo：
- AI 对话对性能敏感
- 需要完善的限流熔断能力
- Triple 协议支持流式响应
```

**2. 为什么选择 MyBatis-Flex 而不是 MyBatis-Plus？**

```
MyBatis-Plus：
- 社区成熟，资料丰富
- Wrapper 链式调用

MyBatis-Flex：
- 更轻量：无反射性能损耗
- SQL 构建更灵活：原生 API
- 多表关联更简洁：无需 XML
- 代码生成器内置

本项目选择 MyBatis-Flex：
- 追求性能极致
- 多表关联查询频繁
```

**3. 为什么选择 LangChain4j 而不是 Spring AI？**

```
Spring AI：
- Spring 官方支持
- 与 Spring 生态集成好
- 但文档较少，功能不够成熟

LangChain4j：
- Java 生态最成熟的 AI 框架
- 支持 OpenAI、Azure、阿里云等多供应商
- 内置 Tool Calling、Memory、Guardrails
- 社区活跃，文档完善

本项目选择 LangChain4j：
- 需要 Tool Calling 能力
- 需要对话记忆管理
- 需要输入输出安全校验
```

**标准回答**：

**1. 自动配置（Auto Configuration）**

Spring Boot 通过 `@SpringBootApplication` 注解启用自动配置：

```java
@SpringBootApplication
public class YuAiCodeAppApplication {
    public static void main(String[] args) {
        SpringApplication.run(YuAiCodeAppApplication.class, args);
    }

```

### 2.4 【追问】Spring 如何解决循环依赖问题？

**问题分析**：循环依赖是 Spring 面试高频题，要深入理解三级缓存机制。

**标准回答**：

**什么是循环依赖**：

```java
@Service
public class ServiceA {
    @Resource
    private ServiceB serviceB;  // A 依赖 B
}

@Service
public class ServiceB {
    @Resource
    private ServiceA serviceA;  // B 依赖 A，形成循环
}
```

**Spring 解决方案：三级缓存**

```
一级缓存（singletonObjects）：
  - 存放完整的 Bean（已初始化完成）
  - Map<String, Object>

二级缓存（earlySingletonObjects）：
  - 存放早期暴露的 Bean（仅实例化，未初始化）
  - 用于解决循环依赖

三级缓存（singletonFactories）：
  - 存放 Bean 工厂对象
  - 用于生成早期 Bean 的代理对象（AOP 场景）
```

**解决流程**：

```
1. 创建 ServiceA
   │
   ▼
2. 实例化 ServiceA（调用构造函数）
   │
   ▼
3. 将 ServiceA 工厂放入三级缓存
   │
   ▼
4. 注入 ServiceB（发现 B 未创建）
   │
   ▼
5. 创建 ServiceB
   │
   ▼
6. 实例化 ServiceB，放入三级缓存
   │
   ▼
7. 注入 ServiceA（从三级缓存获取 A 的早期引用）
   │  └── 三级缓存 → 二级缓存 → 返回早期 Bean
   ▼
8. ServiceB 初始化完成，放入一级缓存
   │
   ▼
9. ServiceA 获取到 ServiceB，初始化完成
   │
   ▼
10. ServiceA 放入一级缓存
```

**为什么需要三级缓存**：

```java
// 如果 ServiceA 需要被 AOP 代理
// 二级缓存存储的是原始对象
// 三级缓存的工厂可以生成代理对象

// singletonFactories 中的工厂
ObjectFactory<?> singletonFactory = () -> {
    Object earlyBean = getEarlyBeanReference(beanName, mbd, bean);
    return earlyBean;  // 可能是代理对象
};
```

**无法解决的循环依赖**：

```java
// 1. 构造器注入的循环依赖（无法解决）
@Service
public class ServiceA {
    private final ServiceB serviceB;
    public ServiceA(ServiceB serviceB) {
        this.serviceB = serviceB;  // 构造时就需要的依赖
    }
}

// 2. prototype 作用域的循环依赖（无法解决）
@Scope("prototype")
@Service
public class ServiceA {
    @Resource
    private ServiceB serviceB;
}
```

**解决构造器注入循环依赖的方法**：

```java
// 使用 @Lazy 延迟加载
@Service
public class ServiceA {
    private final ServiceB serviceB;
    public ServiceA(@Lazy ServiceB serviceB) {
        this.serviceB = serviceB;  // 注入的是代理对象，真正调用时才创建
    }
}
```

**问题分析**：自动配置是 Spring Boot 的核心特性，要理解原理。

```
@SpringBootApplication
```

| `@ConditionalOnProperty` | 配置属性满足条件 |
| `@ConditionalOnWebApplication` | 是 Web 应用 |

### 3.2 什么是 Tool Calling？请结合代码说明

**问题分析**：Tool Calling 是 AI Agent 的核心技术，要深入理解。

**标准回答**：

**Tool Calling 概念**：

Tool Calling（工具调用）让 LLM 能够调用外部函数。LLM 不直接执行代码，而是输出一个"工具调用请求"，由应用层执行后返回结果。

**执行流程**：

```
用户: "帮我生成一个登录页面"
        │
        ▼
┌─────────────────────────────────────┐
│ LLM 推理: "我需要创建 Login.vue 文件" │
│ 输出工具调用请求:                     │
│ {                                    │
│   "name": "writeFile",               │
│   "arguments": {                     │
│     "relativeFilePath": "Login.vue", │
│     "content": "<template>..."       │
│   }                                  │
│ }                                    │
└─────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────┐
│ LangChain4j 解析并执行:              │
│ • 匹配到 FileWriteTool               │
│ • 调用 writeFile() 方法              │
│ • 返回结果: "文件写入成功"            │
└─────────────────────────────────────┘
        │
        ▼
   LLM 继续推理，直到调用 exit 工具
```

**代码实现**：

```java
// FileWriteTool.java - 文件写入工具
```

### 4.1 什么是 SSE？项目中是如何实现的？

> 🔗 **相关问题**：[4.2 Reactor原理](#42-什么是-reactorflux-和-mono-有什么区别) | [4.3 流式保存](#43-流式输出中如何保存-ai-的完整响应) | [11.1 前端SSE接收](#111-前端是如何接收-sse-流式数据的) | [11.4 SSE断线重连](#114-追问前端如何处理-sse-连接中断)

**问题分析**：SSE 是实时通信的重要技术，要理解原理和实现。

**标准回答**：

**SSE（Server-Sent Events）概念**：

SSE 是一种服务器向客户端推送数据的技术，基于 HTTP 长连接，相比 WebSocket 更简单，适合服务器单向推送场景。

**特点对比**：

| 特性 | SSE | WebSocket |
|------|-----|-----------|
| 通信方式 | 单向（服务器→客户端） | 双向 |
| 协议 | HTTP | WS/WSS |
| 断线重连 | 浏览器自动重连 | 需要手动实现 |
| 数据格式 | 文本 | 文本/二进制 |
| 适用场景 | 消息推送、实时日志 | 聊天、游戏 |

**后端实现**：

```java
// AppController.java
@GetMapping(value = "/chat/gen/code", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
@RateLimit(limitType = RateLimitType.USER, rate = 5, rateInterval = 60)
public Flux<ServerSentEvent<String>> chatToGenCode(
        @RequestParam Long appId,
        @RequestParam String message,
        HttpServletRequest request) {

    User loginUser = InnerUserService.getLoginUser(request);
    Flux<String> contentFlux = appService.chatToGenCode(appId, message, loginUser);

    // 将 Flux<String> 转换为 SSE 格式
    return contentFlux
        .map(chunk -> {
            Map<String, String> wrapper = Map.of("d", chunk);
            String jsonData = JSONUtil.toJsonStr(wrapper);
            return ServerSentEvent.<String>builder()
                .data(jsonData)
                .build();
        })
        .concatWith(Mono.just(
            // 发送结束事件
            ServerSentEvent.<String>builder()
                .event("done")
                .data("")
                .build()
        ));
}
```

**关键点解释**：

1. `produces = MediaType.TEXT_EVENT_STREAM_VALUE`：声明返回 SSE 流
2. `Flux<String>`：Reactor 的响应式流，可以异步发送多个数据
3. `ServerSentEvent`：Spring 封装的 SSE 对象
4. `concatWith`：在流结束后追加 done 事件

**SSE 消息格式**：

```
data: {"d":"生成一个"}

data: {"d":"登录页面"}

event: done
data:
```

### 4.2 什么是 Reactor？Flux 和 Mono 有什么区别？

**标准回答**：

**Reactor 简介**：

Reactor 是 Spring 5 响应式编程的核心库，基于 Reactive Streams 规范，提供非阻塞、异步的编程模型。

**Flux 和 Mono 对比**：

| 类型 | 含义 | 数据数量 | 使用场景 |
|------|------|----------|----------|
| Flux<T> | 0-N 个元素的异步序列 | 0到多个 | 列表查询、流式输出 |
| Mono<T> | 0-1 个元素的异步结果 | 0或1个 | 单条查询、保存操作 |

**项目中的使用**：

**常用操作符**：

```java
// map：转换每个元素
flux.map(chunk -> JSONUtil.toJsonStr(chunk))

// filter：过滤元素
flux.filter(chunk -> chunk != null && !chunk.isEmpty())

// doOnNext：处理每个元素（副作用）
flux.doOnNext(chunk -> {

flux.doOnComplete(() -> {
```

### 4.3 流式输出中如何保存 AI 的完整响应？

**问题分析**：流式输出是边生成边返回，但要保存完整内容，需要特殊处理。

**标准回答**：

**问题**：流式输出时，每个 Token 立即发送给前端，但需要收集完整内容后才能保存。

**解决方案**：

```java
// AiCodeGeneratorFacade.java
private Flux<String> processCodeStream(Flux<String> codeStream, 
        CodeGenTypeEnum codeGenType, Long appId) {
    
    // 使用 StringBuilder 收集完整代码
    StringBuilder codeBuilder = new StringBuilder();
    
    return codeStream
        // 每收到一个片段，追加到 StringBuilder
        .doOnNext(chunk -> {
            codeBuilder.append(chunk);
        })
        // 流结束后，保存完整代码
        .doOnComplete(() -> {
            try {
                String completeCode = codeBuilder.toString();
                // 解析代码
                Object parsedResult = CodeParserExecutor.executeParser(completeCode, codeGenType);
                // 保存代码
                CodeFileSaverExecutor.executeSaver(parsedResult, codeGenType, appId);
                log.info("保存成功");
            } catch (Exception e) {
                log.error("保存失败: {}", e.getMessage());
            }
        });
}
```

**关键点**：
1. `doOnNext`：每个数据片段到达时触发，用于收集
2. `doOnComplete`：流结束时触发，用于保存
3. `StringBuilder`：线程不安全，但在同一个流中顺序执行，安全

**异步保存的改进**：

```java
// 使用虚拟线程异步保存，不阻塞响应
.doOnComplete(() -> {
    Thread.startVirtualThread(() -> {
        String completeCode = codeBuilder.toString();
        saveCode(completeCode);
    });
})
```

### 5.3 什么是逻辑删除？项目中是如何实现的？

**问题分析**：逻辑删除是常见的软删除方案，要理解原理和实现。

**标准回答**：

**逻辑删除概念**：

逻辑删除不是真正删除数据，而是将记录标记为"已删除"状态。查询时自动过滤已删除记录。

**优点**：
1. 数据可恢复
2. 保持数据完整性
3. 支持审计追踪

**缺点**：
1. 占用存储空间
2. 需要唯一索引特殊处理
3. 查询性能略受影响

**项目实现**：

```java
// 实体类
@Table("app")
public class App {
    // ...
    
    @Column(value = "isDelete", isLogicDelete = true)
    private Integer isDelete;  // 0: 未删除, 1: 已删除
}
```

**底层原理**：

MyBatis-Flex 自动处理逻辑删除：

```java
// 删除操作：自动转为 UPDATE
appService.removeById(1L);
// 实际执行：UPDATE app SET isDelete = 1 WHERE id = 1

// 查询操作：自动添加条件
appService.getById(1L);
// 实际执行：SELECT * FROM app WHERE id = 1 AND isDelete = 0
```

**源码分析**：

```java
// 删除应用时，同时删除关联数据
@Override
public boolean removeById(Serializable id) {
    if (id == null) {
        return false;
    }
    long appId = Long.parseLong(id.toString());

    // 先删除关联的对话历史
    try {
        chatHistoryService.deleteByAppId(appId);
    } catch (Exception e) {
        log.error("删除对话历史失败：{}", e.getMessage());
    }

    // 删除应用（逻辑删除）
    return super.removeById(id);
}
```

### 5.4 【追问】项目中使用的雪花算法 ID 有什么特点？

**问题分析**：分布式 ID 生成是后端常见问题，要理解主流方案的优劣。

**标准回答**：

**分布式 ID 方案对比**：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| UUID | 简单、无依赖 | 无序、太长、无业务含义 | 非主键场景 |
| 数据库自增 | 简单、有序 | 单点瓶颈、分库分表问题 | 单库小规模 |
| Redis 自增 | 高性能、有序 | 依赖 Redis、需要持久化 | 中等规模 |
| 雪花算法 | 有序、高性能、分布式 | 依赖时钟 | 大规模分布式 |
| 号段模式 | 高性能、连续 | 需要数据库表 | 超大规模 |

**雪花算法原理**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    Snowflake ID 结构 (64 bit)                    │
├────────────┬─────────────┬─────────────┬────────────────────────┤
│  符号位(1)  │  时间戳(41)  │ 机器ID(10)  │    序列号(12)          │
│    0       │ 毫秒级时间戳  │ 5位数据中心  │   同一毫秒内的序列      │
│            │ (约69年)     │ 5位机器ID    │   (每毫秒4096个)        │
└────────────┴─────────────┴─────────────┴────────────────────────┘

ID 示例：1234567890123456789
解析：
- 时间戳部分：确定 ID 生成时间
- 机器 ID：确定是哪台机器生成
- 序列号：同一毫秒内的序号
```

**项目中的实现**：

### 6.2 什么是缓存穿透、缓存击穿、缓存雪崩？如何解决？

**问题分析**：这是 Redis 面试必考题，要能解释概念并给出解决方案。

**标准回答**：

**1. 缓存穿透**

**问题**：查询不存在的数据，缓存和数据库都没有，每次请求都穿透到数据库。

**攻击场景**：恶意请求 id=-1，绕过缓存直接查询数据库。

**解决方案**：

```java
// 方案1：缓存空值
@GetMapping("/app/{id}")
public App getApp(@PathVariable Long id) {
    String key = "app:" + id;
    App app = redis.get(key);
    
    if (app != null) {
        return app;
    }
    
    app = appMapper.selectById(id);
    
    if (app == null) {
        // 缓存空值，设置较短过期时间
        redis.set(key, "NULL", 60);
        return null;
    }
    
    redis.set(key, app, 1800);
    return app;
}

// 方案2：布隆过滤器
// 在项目启动时，将所有有效 id 加载到布隆过滤器
// 查询前先判断 id 是否可能存在
```

**2. 缓存击穿**

**问题**：某个热点 key 过期，大量请求同时查询数据库。

**解决方案**：

```java
// 方案1：互斥锁
public App getAppWithLock(Long id) {
    String key = "app:" + id;
    App app = redis.get(key);
    
    if (app != null) {
        return app;
    }
    
    // 获取分布式锁
    String lockKey = "lock:app:" + id;
    try {
        boolean locked = redis.tryLock(lockKey, 10);
        if (locked) {
            // 再次检查缓存（双重检查）
            app = redis.get(key);
            if (app != null) {
                return app;
            }
            
            // 查询数据库
            app = appMapper.selectById(id);
            redis.set(key, app, 1800);
            return app;
        } else {
            // 等待并重试
            Thread.sleep(50);
            return getAppWithLock(id);
        }
    } finally {
        redis.unlock(lockKey);
    }
}

// 方案2：逻辑过期（不设置 TTL，后台异步更新）
```

**3. 缓存雪崩**

**问题**：大量 key 同时过期，或 Redis 宕机，所有请求打到数据库。

**解决方案**：

```java
// 方案1：随机过期时间
int randomExpire = 1800 + new Random().nextInt(300);  // 30-35分钟
redis.set(key, value, randomExpire);

// 方案2：缓存预热
// 项目启动时，预加载热点数据到缓存

// 方案3：Redis 高可用
// 主从复制 + 哨兵模式
```

**项目中的应对**：

```java
// RedisCacheManagerConfig.java
RedisCacheConfiguration defaultConfig = RedisCacheConfiguration.defaultCacheConfig()
    .entryTtl(Duration.ofMinutes(30))
    .disableCachingNullValues();  // 防止缓存穿透

// 针对不同缓存设置不同过期时间，避免雪崩

.withCacheConfiguration("user_info", defaultConfig.entryTtl(Duration.ofMinutes(60)))
```

### 6.3 项目中如何保证缓存一致性？

**问题分析**：缓存与数据库一致性是分布式系统经典问题。

**标准回答**：

**常见策略对比**：

| 策略 | 描述 | 一致性 | 性能 |
|------|------|--------|------|
| 先更新数据库，再更新缓存 | 更新后立即刷新缓存 | 较好 | 一般 |
| 先删除缓存，再更新数据库 | 更新时删除缓存 | 较差 | 较好 |
| 先更新数据库，再删除缓存 | 推荐，延迟双删 | 较好 | 较好 |

**本项目策略**：先更新数据库，再删除缓存

```java
// 更新应用时，删除相关缓存
@PostMapping("/update")
public BaseResponse<Boolean> updateApp(@RequestBody AppUpdateRequest request) {
    // 1. 更新数据库
    App app = new App();
    app.setId(request.getId());
    app.setAppName(request.getAppName());
    app.setEditTime(LocalDateTime.now());
    boolean result = appService.updateById(app);
    
    // 2. 删除缓存（由 @CacheEvict 处理）
    // 或者手动删除
    // cacheManager.getCache("good_app_page").clear();
    
    return ResultUtils.success(result);
}
```

**延迟双删**：解决并发问题

```java
public void updateApp(App app) {
    // 1. 删除缓存
    cacheManager.getCache("app").evict(app.getId());
    
    // 2. 更新数据库
    appMapper.updateById(app);
    
    // 3. 延迟后再删除缓存（解决并发读取旧数据问题）
    Thread.startVirtualThread(() -> {
        try {
            Thread.sleep(500);  // 延迟 500ms
            cacheManager.getCache("app").evict(app.getId());
        } catch (InterruptedException e) {
            log.error("延迟删除缓存失败", e);
        }
    });
}
```

**面试加分点**：

1. **Canal 监听 binlog**：可以监听 MySQL binlog 自动更新缓存
2. **消息队列**：更新数据库后发消息，消费者负责删除缓存
3. **最终一致性**：分布式系统中，追求最终一致性而非强一致性

### 6.4 【追问】如何实现分布式锁？项目中用到了吗？

**问题分析**：分布式锁是高频面试题，要理解原理和实现细节。

**标准回答**：

**为什么需要分布式锁**：

```
单机环境：
- 使用 synchronized 或 ReentrantLock 即可

分布式环境：
- 多个服务实例，JVM 级别的锁无效
- 需要跨进程、跨机器的锁
```

**分布式锁实现方案对比**：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| Redis SETNX | 简单、性能高 | 需要处理超时、续期 | 一般场景 |
| Redisson | 功能完善、可重入 | 依赖 Redis | 推荐使用 |
| ZooKeeper | 可靠性高 | 性能较低、复杂 | 强一致性场景 |
| etcd | 性能好、可靠 | 学习成本 | 云原生场景 |

**Redis 基础实现**：

```java
// 加锁
public boolean tryLock(String key, String value, long expireTime) {
    // SET key value NX EX expireTime
    // NX: 不存在才设置
    // EX: 设置过期时间（秒）
    return redisTemplate.opsForValue()
        .setIfAbsent(key, value, expireTime, TimeUnit.SECONDS);
}

// 解锁（Lua 脚本保证原子性）
public boolean releaseLock(String key, String value) {
    String script =
        "if redis.call('get', KEYS[1]) == ARGV[1] then " +
        "    return redis.call('del', KEYS[1]) " +
        "else " +
        "    return 0 " +
        "end";
    return redisTemplate.execute(
        new DefaultRedisScript<>(script, Long.class),
        Collections.singletonList(key),
        value
    ) == 1;
}
```

**Redisson 实现（推荐）**：

```java
// 项目中使用 Redisson
@Configuration
public class RedissonConfig {

    @Bean
    public RedissonClient redissonClient() {
        Config config = new Config();
        config.useSingleServer()
            .setAddress("redis://localhost:6379")
            .setPassword("password")
            .setDatabase(0);
        return Redisson.create(config);
    }
}

// 使用示例
@Service
public class AppServiceImpl {

    @Resource
    private RedissonClient redissonClient;

    public void deployApp(Long appId) {
        // 获取分布式锁
        String lockKey = "lock:deploy:" + appId;
        RLock lock = redissonClient.getLock(lockKey);

        try {
            // 尝试获取锁，最多等待 10 秒，锁自动释放时间 30 秒
            boolean locked = lock.tryLock(10, 30, TimeUnit.SECONDS);
            if (!locked) {
                throw new BusinessException(ErrorCode.OPERATION_ERROR, "其他用户正在部署，请稍后再试");
            }

            // 执行部署逻辑
            doDeploy(appId);

        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new BusinessException(ErrorCode.SYSTEM_ERROR);
        } finally {
            // 只释放自己持有的锁
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

**Redisson 分布式锁的特性**：

```
1. 可重入：同一个线程可以多次获取同一把锁
2. 自动续期：看门狗机制，自动延长锁过期时间
3. 公平锁：支持公平获取锁
4. 读写锁：支持读写锁分离
5. 红锁：多 Redis 节点的高可用锁
```

### 6.5 【追问】Redis 是如何持久化的？各有什么优缺点？

**问题分析**：Redis 持久化是保障数据安全的重要机制。

**标准回答**：

**RDB（Redis Database）**：

```
原理：定时将内存中的数据快照保存到磁盘

触发方式：
1. SAVE：阻塞主进程，直接保存
2. BGSAVE：fork 子进程保存，不阻塞主进程

配置示例：
save 900 1      # 900秒内至少1次修改触发
save 300 10     # 300秒内至少10次修改触发
save 60 10000   # 60秒内至少10000次修改触发

文件格式：
dump.rdb（二进制压缩格式）
```

**AOF（Append Only File）**：

```
原理：记录每次写操作命令，恢复时重放

同步策略：
1. always：每次写入都同步，最安全但最慢
2. everysec：每秒同步一次（推荐）
3. no：由操作系统决定，最快但可能丢数据

文件格式：
appendonly.aof（文本格式，可读）
```

**对比**：

| 特性 | RDB | AOF |
|------|-----|-----|
| 数据安全性 | 可能丢失几分钟数据 | 最多丢失 1 秒数据 |
| 文件大小 | 小（压缩） | 大（命令日志） |
| 恢复速度 | 快 | 慢（需要重放） |
| 系统资源 | fork 时消耗内存 | 持续写入磁盘 |
| 适用场景 | 允许少量丢失、快速恢复 | 数据安全要求高 |

**混合持久化（Redis 4.0+）**：

```
aof-use-rdb-preamble yes

原理：
1. AOF 重写时，先写入 RDB 格式的快照
2. 然后追加增量 AOF 命令

优点：
- 结合两者优点
- 快速恢复 + 低数据丢失
```

**项目中的配置建议**：

```bash
# redis.conf
# 开启 AOF
appendonly yes

# 每秒同步（推荐）
appendfsync everysec

# 开启混合持久化
aof-use-rdb-preamble yes

# AOF 重写触发条件
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

### 7.1 项目是如何实现限流的？为什么要用 Redisson？

> 🔗 **相关问题**：[7.2 令牌桶算法](#72-什么是令牌桶算法和漏桶算法有什么区别) | [7.3 窗口算法对比](#73-追问固定窗口和滑动窗口限流有什么区别) | [6.1 Redis使用场景](#61-项目中-redis-有哪些使用场景) | [6.4 分布式锁](#64-追问如何实现分布式锁项目中用到了吗)

**问题分析**：限流是保护系统的重要手段，要理解原理和实现。

**标准回答**：

**限流的必要性**：

1. 保护系统不被突发流量打垮
2. 防止恶意攻击
3. 保证服务公平性

**本项目限流实现**：

```java
// RateLimit.java - 自定义注解
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RateLimit {
    String key() default "";              // 限流 key 前缀
    RateLimitType limitType() default RateLimitType.API;  // 限流类型
    int rate() default 10;               // 时间窗口内允许的请求数
    int rateInterval() default 60;       // 时间窗口（秒）
    String message() default "请求过于频繁";  // 提示消息
}

// RateLimitType.java - 限流类型
public enum RateLimitType {
    API,   // 接口级别：按方法签名限流
    USER,  // 用户级别：按用户 ID 限流
    IP     // IP 级别：按客户端 IP 限流
}
```

**切面实现**：

```java
// RateLimitAspect.java
@Aspect
@Component
public class RateLimitAspect {

    @Resource
    private RedissonClient redissonClient;

    @Before("@annotation(rateLimit)")
    public void doBefore(JoinPoint point, RateLimit rateLimit) {
        String key = generateRateLimitKey(point, rateLimit);

        // 使用 Redisson 的分布式限流器
        RRateLimiter rateLimiter = redissonClient.getRateLimiter(key);
        rateLimiter.expire(Duration.ofHours(1));  // 1 小时后过期

        // 设置令牌桶参数
        rateLimiter.trySetRate(
            RateType.OVERALL,           // 全局限流
            rateLimit.rate(),           // 令牌数量
            rateLimit.rateInterval(),   // 时间窗口
            RateIntervalUnit.SECONDS
        );

        // 尝试获取令牌
        if (!rateLimiter.tryAcquire(1)) {
            throw new BusinessException(ErrorCode.TOO_MANY_REQUEST, rateLimit.message());
        }
    }

    private String generateRateLimitKey(JoinPoint point, RateLimit rateLimit) {
        StringBuilder keyBuilder = new StringBuilder("rate_limit:");

        switch (rateLimit.limitType()) {
            case API:
                // 接口级别：类名.方法名
                Method method = ((MethodSignature) point.getSignature()).getMethod();
                keyBuilder.append("api:")
                    .append(method.getDeclaringClass().getSimpleName())
                    .append(".").append(method.getName());
                break;
            case USER:
                // 用户级别：用户 ID
                User loginUser = InnerUserService.getLoginUser(request);
                keyBuilder.append("user:").append(loginUser.getId());
                break;
            case IP:
                // IP 级别：客户端 IP
                keyBuilder.append("ip:").append(getClientIP());
                break;
        }

        return keyBuilder.toString();
    }
}
```

### 7.2 什么是令牌桶算法？和漏桶算法有什么区别？

**问题分析**：限流算法是面试常考点，要能画图说明原理。

**标准回答**：

**令牌桶算法**：

```
                ┌─────────────────┐
                │   令牌生成器    │
                │  (固定速率生成) │
                └────────┬────────┘
                         │
                         ▼ 放入令牌
                ┌─────────────────┐
                │     令牌桶      │
                │   ┌───┬───┬───┐ │
                │   │ ● │ ● │   │ │  最多 N 个令牌
                │   └───┴───┴───┘ │
                └────────┬────────┘
                         │
            ┌────────────┼────────────┐
            │            │            │
            ▼            ▼            ▼
        ┌───────┐   ┌───────┐   ┌───────┐
        │ 请求1 │   │ 请求2 │   │ 请求3 │
        │ 取令牌│   │ 等待  │   │ 拒绝  │
        └───────┘   └───────┘   └───────┘
```

**特点**：
- 以固定速率生成令牌放入桶中
- 请求到达时从桶中取令牌
- 桶满时令牌溢出
- 无令牌时请求被拒绝或等待

**漏桶算法**：

```
        请求流入
           │
           ▼
    ┌──────────────┐
    │    漏桶      │
    │  ┌────────┐  │
    │  │ 水 水 水│  │  缓冲队列
    │  └────────┘  │
    │      │       │
    │      ▼       │
    │   [小孔]     │  固定速率流出
    └──────┼───────┘
           │
           ▼
        请求流出
```

**特点**：
- 请求先进入队列
- 以固定速率处理请求
- 队列满时拒绝新请求

**对比**：

| 特性 | 令牌桶 | 漏桶 |
|------|--------|------|
| 突发流量 | 允许一定突发 | 严格限制 |
| 实现复杂度 | 较高 | 较低 |
| 适用场景 | API 限流 | 流量整形 |
| 请求延迟 | 低 | 可能较高 |

**本项目选择令牌桶的原因**：

AI 对话请求有时会突发（用户连续提问），令牌桶允许短时突发，用户体验更好。

### 7.3 【追问】固定窗口和滑动窗口限流有什么区别？

**问题分析**：限流算法细节考察，要理解边界情况。

**标准回答**：

**固定窗口算法**：

```
时间轴：|----60s----|----60s----|----60s----|
计数：  |    10     |    10     |    10     |

问题：边界突发
        ↓
在 59s 时来了 10 个请求，61s 时又来了 10 个请求
虽然每分钟都是 10 个，但在 59s-61s 这 2 秒内来了 20 个请求

┌─────────────────────────────────┐
│  第1个窗口         第2个窗口     │
│  [...58][59]  [60][61]...       │
│    10请求      10请求            │
│                                  │
│  59s-61s 之间实际有 20 个请求！  │
└─────────────────────────────────┘
```

**滑动窗口算法**：

```
时间轴：不断滑动的窗口
当前时间点向前推 60s 作为窗口

┌─────────────────────────────────┐
│  真正的 60s 窗口                 │
│  [30][31][32]...[89][90]        │
│  滑动窗口始终是最近 60s          │
└─────────────────────────────────┘

优点：更精确的限流
缺点：需要记录每个请求的时间戳，内存占用高
```

**代码实现对比**：

```java
// 固定窗口（Redis 实现）
public boolean allowRequest(String key, int limit, int windowSeconds) {
    long count = redisTemplate.opsForValue().increment(key);
    if (count == 1) {
        redisTemplate.expire(key, windowSeconds, TimeUnit.SECONDS);
    }
    return count <= limit;
}

// 滑动窗口（Redis ZSet 实现）
public boolean allowRequestSlidingWindow(String key, int limit, int windowSeconds) {
    long now = System.currentTimeMillis();
    long windowStart = now - windowSeconds * 1000;

    // 移除窗口外的请求
    redisTemplate.opsForZSet().removeRangeByScore(key, 0, windowStart);

    // 统计当前窗口内的请求数
    long count = redisTemplate.opsForZSet().count(key, windowStart, now);

    if (count < limit) {
        // 添加当前请求
        redisTemplate.opsForZSet().add(key, UUID.randomUUID().toString(), now);
        return true;
    }
    return false;
}
```

**本项目使用的令牌桶 vs 滑动窗口**：

```
令牌桶优势：
1. 允许一定程度的突发流量
2. 实现简单，Redisson 内置
3. 性能好，不需要记录每个请求

滑动窗口优势：
1. 更精确的控制
2. 绝不会有突发问题

本项目选择令牌桶：
- AI 对话场景允许短时突发
- 用户连续提问是正常行为
- 性能优先
```

**RBAC 模型**：

**本项目实现**：

**1. 角色定义**：

```java
    private final String desc;
```

**2. 权限注解**：

```java
@Target(ElementType.METHOD)
```

**3. AOP 拦截**：

```java
@Aspect
@Component
public class AuthInterceptor {

    @Around("@annotation(authCheck)")
    public Object doInterceptor(ProceedingJoinPoint joinPoint, AuthCheck authCheck) throws Throwable {
        String mustRole = authCheck.mustRole();
        
        // 获取当前登录用户
        HttpServletRequest request = getRequest();
        User loginUser = InnerUserService.getLoginUser(request);
        
        UserRoleEnum mustRoleEnum = UserRoleEnum.getEnumByValue(mustRole);
        
        // 不需要权限，直接放行
        if (mustRoleEnum == null) {
            return joinPoint.proceed();
        }
        
        // 获取用户角色
        UserRoleEnum userRoleEnum = UserRoleEnum.getEnumByValue(
            loginUser.getUserRole().toLowerCase()
        );
        
        // 没有权限
        if (userRoleEnum == null) {
            throw new BusinessException(ErrorCode.NO_AUTH_ERROR);
        }
        
        // 要求管理员权限
        if (UserRoleEnum.ADMIN.equals(mustRoleEnum) && 
            !UserRoleEnum.ADMIN.equals(userRoleEnum)) {
            throw new BusinessException(ErrorCode.NO_AUTH_ERROR);
        }
        
        return joinPoint.proceed();
    }
}
```

**4. 使用示例**：

```java

}
```

### 8.3 密码是如何加密存储的？

```java
public boolean matches(String rawPassword, String encodedPassword) {
```

### 8.5 【追问】如何防止 CSRF 攻击？

**问题分析**：Web 安全是必考知识点。

**标准回答**：

**CSRF 攻击原理**：

```
1. 用户登录了银行网站 bank.com
2. 用户访问了恶意网站 evil.com
3. evil.com 包含：<img src="bank.com/transfer?to=hacker&amount=10000">
4. 浏览器自动带上 bank.com 的 Cookie
5. 银行服务器验证通过，转账成功
```

**防御措施**：

**1. CSRF Token**：

```java
// Spring Security 默认开启 CSRF
// 每个表单需要包含 CSRF Token
<form>
    <input type="hidden" name="_csrf" value="${_csrf.token}"/>
</form>

// 前端存储 Token
axios.defaults.headers.common['X-CSRF-TOKEN'] = csrfToken;
```

**2. SameSite Cookie**：

```java
// Cookie 设置 SameSite 属性
response.setHeader("Set-Cookie",
    "sessionId=xxx; SameSite=Strict; Secure; HttpOnly");

// SameSite 值：
// - Strict：完全禁止第三方 Cookie
// - Lax：允许 GET 请求携带（默认）
// - None：允许所有（需要配合 Secure）
```

**3. 验证 Referer 头**：

```java
// 检查请求来源
public boolean isValidReferer(HttpServletRequest request) {
    String referer = request.getHeader("Referer");
    if (referer == null) {
        return false;
    }
    return referer.startsWith("https://mydomain.com");
}
```

**本项目的情况**：


### 11.3 【追问】Vue 3 的响应式原理是什么？

**问题分析**：前端框架核心原理考察。

**标准回答**：

**Vue 2 vs Vue 3 响应式对比**：

```
Vue 2 (Object.defineProperty)：
- 无法检测对象属性的添加/删除
- 无法检测数组索引变化
- 需要使用 $set/$delete

Vue 3 (Proxy)：
- 可以检测对象属性的添加/删除
- 可以检测数组索引变化
- 性能更好，惰性代理
```

**Proxy 实现原理**：

```javascript
// Vue 3 响应式核心
function reactive(target) {
  return new Proxy(target, {
    get(target, key, receiver) {
      // 收集依赖
      track(target, key)
      const result = Reflect.get(target, key, receiver)
      // 深层响应式
      if (typeof result === 'object') {
        return reactive(result)
      }
      return result
    },

    set(target, key, value, receiver) {
      const oldValue = target[key]
      const result = Reflect.set(target, key, value, receiver)
      if (oldValue !== value) {
        // 触发更新
        trigger(target, key)
      }
      return result
    },

    deleteProperty(target, key) {
      const result = Reflect.deleteProperty(target, key)
      trigger(target, key)
      return result
    }
  })
}
```

**项目中的重连机制**：

```typescript
// useAppChat.ts
class SseConnection {
  private eventSource: EventSource | null = null
  private reconnectAttempts = 0
  private maxReconnectAttempts = 5
  private reconnectDelay = 1000  // 初始延迟 1 秒

  connect(url: string) {
    this.eventSource = new EventSource(url)

    this.eventSource.onopen = () => {
      console.log('SSE 连接建立')
      this.reconnectAttempts = 0  // 重置重连次数
      this.reconnectDelay = 1000  // 重置延迟
    }

    this.eventSource.onerror = (error) => {
      console.error('SSE 连接错误:', error)

      if (this.eventSource?.readyState === EventSource.CLOSED) {
        // 连接已关闭，尝试重连
        this.reconnect()
      }
    }
  }

  private reconnect() {
    if (this.reconnectAttempts >= this.maxReconnectAttempts) {
      message.error('AI 服务连接失败，请刷新页面重试')
      return
    }

    this.reconnectAttempts++
    const delay = this.reconnectDelay * Math.pow(2, this.reconnectAttempts - 1)
    // 指数退避：1s, 2s, 4s, 8s, 16s

    console.log(`尝试重连 (${this.reconnectAttempts}/${this.maxReconnectAttempts})，${delay}ms 后`)

    setTimeout(() => {
      this.connect(this.lastUrl)
    }, delay)
  }

  disconnect() {
    this.eventSource?.close()
    this.eventSource = null
  }
}
```

```typescript
function renderCodeInChunks(code: string, chunkSize = 100) {
  const lines = code.split('\n')
  const chunks = []

  for (let i = 0; i < lines.length; i += chunkSize) {
    chunks.push(lines.slice(i, i + chunkSize))
  }

  // 先渲染第一块
  renderedLines.value = chunks[0]

  // 延迟渲染后续块
  chunks.slice(1).forEach((chunk, index) => {

    }, (index + 1) * 100)
  })
}
```

```typescript
class VisualEditor {
  private iframe: HTMLIFrameElement

  init(iframe: HTMLIFrameElement) {
    this.iframe = iframe
    this.injectScript()

      document.addEventListener('mouseover', (e) => {
        e.target.style.outline = '2px solid #667eea'
      })
      
      document.addEventListener('mouseout', (e) => {
        e.target.style.outline = 'none'
      })
      
      // 点击选中元素
      document.addEventListener('click', (e) => {
        e.preventDefault()
        const elementInfo = {
          tagName: e.target.tagName,
          id: e.target.id,
          className: e.target.className,
          textContent: e.target.textContent?.substring(0, 50),
          selector: getSelector(e.target)
        }

        window.parent.postMessage({ type: 'element-selected', data: elementInfo }, '*')
      })
    `
    doc.body.appendChild(script)
  }
}
```
